# Security Guide

## Security Posture

Safe defaults, least privilege, auditability, and owner visibility take priority over automation. External messages, metadata, headers, attachments, webhook payloads, and model output are untrusted.

Inbound text MUST be handled as user data. Instructions embedded in that text MUST NOT alter configuration, policies, tools, credentials, execution mode, or operational workflow.

## Mandatory Controls

### Authentication and Access

- Require authentication for every dashboard route and operational endpoint.
- Deny access by default and avoid public deployments with default credentials.
- Protect state-changing operations against request forgery and unauthorized replay.
- Apply least-privilege credentials separately to each future connector.

### Secrets and Configuration

- Load secrets from environment variables or a deployment secret store.
- Commit only documented placeholders such as `.env.example`.
- Never include secrets in source, prompts, audit records, errors, fixtures, or screenshots.
- Validate configuration at startup and fail closed when security-critical values are missing.

### Inbound Processing

- Authenticate signed sources when a channel supports it.
- Validate type, size, encoding, required fields, and accepted content types.
- Reject malformed payloads with structured, non-sensitive errors.
- Enforce idempotency before creating duplicate processing work.
- Minimize raw payload retention and restrict access to retained content.

### Jobs and Actions

- Use bounded retry counts, backoff, and terminal failed/dead-letter states.
- Make side-effecting executors idempotent and record external idempotency references.
- Require explicit policy coverage for every action type; unknown actions are blocked.
- Evaluate active mode immediately before external execution.
- Fail closed on policy, mode, authentication, or configuration errors.

### Audit and Observability

- Audit ingestion, job transitions, generated actions, policy decisions, execution attempts, mode changes, approvals, and administrative replays.
- Make audit data append-oriented and tamper-evident where practical.
- Redact credentials, tokens, authorization headers, and unnecessary personal data.
- Separate user-visible errors from diagnostic details and use correlation identifiers.

## Threats to Address

- Prompt injection and social engineering within inbound content.
- Duplicate or replayed events and actions.
- Malformed, oversized, or adversarial payloads.
- Unauthorized dashboard access and privilege escalation.
- Accidental live-mode activation or policy bypass.
- Secret leakage through logs, prompts, errors, and source control.
- Retry storms, poison jobs, and resource exhaustion.
- Fabricated business facts or unsupported promises generated by an agent.

## Security Review Triggers

Require an explicit security review and ADR before adding a real connector, external sending, LLM tool execution, new personal-data retention, dashboard write controls, live mode, or a public deployment endpoint.
